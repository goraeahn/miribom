# 미리봄(Miribom) — 개발자 인계 문서

작성 목적: 본 결과물을 개발자에게 인계하여 **GitHub 공개**와 **서버 배포**를 진행하기 위한 설명·고려사항·인계사항 정리.
대상 산출물: `미리봄.html` (배포 버전 **미리봄 1.0**, 주식회사 코드넛 / codnut.com)

---

## 1. 한 줄 요약

마크다운(.md) 원고를 **빌드 과정 없이** 브라우저에서 책처럼 보여 주는 **단일 HTML 파일** 뷰어입니다. 프레임워크·번들러·백엔드가 필요 없고, 외부 런타임 의존성은 **Google Fonts 하나**뿐입니다.

## 2. 핵심 특징

- **단일 파일·무빌드**: `미리봄.html` 하나에 HTML·CSS·JavaScript가 모두 들어 있습니다. 더블클릭으로 바로 실행됩니다.
- **자체 마크다운 파서**: 외부 마크다운 라이브러리를 쓰지 않고 직접 구현했습니다(아래 3·6장). YAML 프런트매터, `:::` 의미 블록(표지·판권·목차 등), 각주, 표, 코드펜스, 장면 전환 등을 지원합니다.
- **책 조판**: CSS 다단(multi-column)으로 페이지를 나누고, 표지·장 시작 새 쪽·여백·쪽 번호·목차를 자동 구성합니다.
- **뷰어 설정**: 글꼴(명조/고딕)·크기·행간·여백·판형·보기(한 면/두 면)·테마(주간/세피아/야간)·각주 표시 방식 등.
- **첫 화면(랜딩)**: 배경 입자 네트워크 애니메이션(캔버스), 제목 타이핑 효과, 밑줄 노트형 업로드 박스.
- **저장·통신 없음**: localStorage·서버 호출이 없습니다. 모든 동작은 클라이언트에서 끝나며, 설정은 새로고침 시 초기화됩니다.

## 3. 기술 구조

- **구성**: 표준 HTML 문서 1개. `<head>`에 인라인 `<style>`, `<body>` 끝에 인라인 `<script>`(즉시실행함수 IIFE). 빌드 산출물이 아니라 원본 그대로가 배포본입니다.
- **외부 자원**: `fonts.googleapis.com` / `fonts.gstatic.com`에서 **Noto Serif KR**, **Noto Sans KR**를 불러옵니다(OFL 라이선스). 그 외 CDN·스크립트·이미지 외부 요청은 없습니다. 폰트를 못 불러오면 시스템 명조/고딕으로 대체되어 동작은 유지됩니다.
- **렌더 파이프라인**:
  1. `assemble(md)` — 마크다운을 파싱해 본문 HTML·표지/속표지/판권 등 프런트매터·각주·목차 랜드마크·메타데이터를 만듭니다(순수 함수).
  2. `buildFlow()` / `composeFlow()` — 설정에 맞춰 최종 흐름(flow) HTML을 조립합니다.
  3. `paginate()` — CSS 다단으로 배치한 뒤 한 번 측정해 페이지 수와 장(章) 시작 위치를 계산합니다.
  4. `render()` — `viewport.scrollLeft`로 해당 페이지를 보여 줍니다(가로 스크롤 방식).
- **페이지 이동**: 초기에는 거대 요소를 `transform`으로 옮겼으나, 큰 원고에서 수십만 px 폭의 GPU 레이어가 생겨 멈추는 문제가 있어 **뷰포트 가로 스크롤** 방식으로 변경했습니다.
- **단일 진입점(seam)**: 원고 로딩은 모두 `loadManuscript(md)` 한 곳을 거칩니다. 추후 서버·토큰 기반 원고 로딩을 붙일 확장 지점입니다.
- **내장 샘플**: 사용 설명서가 `<script type="text/markdown" id="sample-doc">`에 들어 있어, "사용 설명서 보기"로 즉시 로드됩니다.

## 5. 로컬 실행

별도 서버 없이 `미리봄.html`을 브라우저로 열면 됩니다(`file://`). 단, Google Fonts를 받으려면 인터넷이 필요합니다. 마우스로 .md를 끌어다 놓거나 업로드 박스를 눌러 원고를 엽니다.

## 6. 개발자에게 설명할 핵심 포인트

- **빌드/의존성이 없다**: npm·webpack·React 등이 없습니다. 파일을 열거나 정적 호스팅에 올리면 끝입니다.
- **마크다운 규격이 핵심 자산이다**: 표준 CommonMark가 아니라, 책 조판을 위한 **자체 규격**입니다. 규격 정의는 `미리봄-전면수정-프롬프트.md`, 동작 기준은 파서 함수 `assemble()`, 적합성 기준은 검증 픽스처입니다. 수정 시 이 셋을 함께 봐야 합니다.
- **조판은 브라우저 CSS 다단에 의존한다**: 정확한 쪽나눔(줄 단위 분리)을 브라우저 레이아웃에 맡깁니다. 장점은 단순함, 단점은 아주 큰 원고에서 전체를 한 번에 배치하는 비용(아래 7장).
- **상태는 메모리에만 있다**: 새로고침하면 설정·원고가 사라집니다(영속화 미구현).
- **확장 지점은 `loadManuscript(md)`**: 서버에서 원고를 받아오려면 이 함수에 연결하면 됩니다.

- **라이선스(LICENSE) 선정**: 코드 본문(파서·뷰어)은 자체 구현입니다. 공개 정책에 맞는 라이선스를 정하세요(예: MIT, 또는 소스 비공개 정책이면 사내 보관).
- **서드파티 고지**:
  - **아이콘** — 업로드/특성 아이콘은 Feather 계열의 단색 스트로크 아이콘 형태입니다. Feather Icons는 MIT입니다. 동일 디자인을 사용했다면 README에 **MIT 저작자 표시**를 넣는 것이 안전합니다(정확한 출처/문구는 개발자가 확정).
  - **폰트** — Noto Serif KR / Noto Sans KR(SIL Open Font License). 핫링크 사용은 무방하나, 오프라인 배포 시 셀프호스팅하려면 OFL 고지를 포함하세요.
- **민감 콘텐츠 — 중요**: `메아리_검증픽스처.md`에는 **저작자 본인의 창작 소설 「메아리」 원고**가 들어 있습니다. 공개 리포지토리에 올리면 미발표 창작물이 그대로 노출됩니다. **공개 전 반드시** (a) 픽스처를 비공개로 두거나, (b) 동일한 기능을 모두 포함하는 **중립적·공개 가능한 원고**로 교체하세요. 내장 샘플(사용 설명서)은 공개에 문제없습니다.
- **데모(GitHub Pages)**: 정적 단일 파일이라 Pages·Netlify·Vercel 어디든 그대로 올라가며 데모가 즉시 동작합니다. `index.html`로 두면 별도 설정이 거의 필요 없습니다.
- **README**: 본 문서의 1·2·5장과 스크린샷, 데모 링크, 라이선스/고지를 묶어 README로 만들면 됩니다.

## 8. 서버 업로드 시 고려사항

- **정적 호스팅이면 충분**: 백엔드 불필요. Nginx/Apache, S3+CloudFront, GitHub Pages, Netlify, Vercel, 또는 codnut.com 하위 경로 등 어디든 가능합니다.
- **파일명**: 한글 파일명 `미리봄.html`은 일부 서버·URL에서 퍼센트 인코딩/인코딩 깨짐을 유발할 수 있습니다. **`index.html`** 또는 `miribom.html` 같은 ASCII 이름으로 올리길 권장합니다.
- **인코딩(UTF-8)**: 파일은 UTF-8입니다. 서버가 `Content-Type: text/html; charset=utf-8`로 응답하게 하세요. 빌드/이관 과정에서 CP949(EUC-KR)로 재저장되지 않도록 주의(한글 깨짐 원인 1순위).
- **HTTPS 권장**: Google Fonts가 https라서, 사이트도 https로 서빙해야 혼합 콘텐츠 차단을 피합니다.
- **CSP(콘텐츠 보안 정책) 적용 시**: 스크립트·스타일이 **인라인**입니다. 엄격한 CSP를 건다면 `script-src`/`style-src`에 `'unsafe-inline'`을 허용하거나, 스크립트·스타일을 외부 파일로 분리해 해시/논스를 부여해야 합니다. 폰트용으로 `style-src https://fonts.googleapis.com`, `font-src https://fonts.gstatic.com`도 필요합니다.
- **오프라인/폐쇄망**: Google Fonts 접근이 막히면 시스템 폰트로 대체되어 동작은 하나 모양이 달라집니다. 폰트 고정이 필요하면 woff2를 내려받아 셀프호스팅하고 `<link>`를 로컬 `@font-face`로 교체하세요.
- **캐싱**: 단일 파일이라 배포가 간단합니다. 업데이트 반영을 위해 파일명 버저닝(`miribom.v1.html`) 또는 적절한 `Cache-Control`을 고려하세요.

## 9. 알려진 한계 · 작업 인계사항(중요)

1. **대용량 원고 조판 비용** — 정확한 쪽나눔을 위해 CSS 다단으로 **책 전체를 한 번에 배치**합니다. 수백 KB 이상 원고에서는 첫 조판이 느릴 수 있습니다. 완화 조치는 적용했습니다(거대 GPU 레이어 제거를 위한 스크롤 방식 전환, 불필요한 `break-inside:avoid` 완화, 큰 원고 로딩 안내). **근본 해결(장 단위 지연/청크 페이지네이션)은 미구현**이며, 대형 원고가 주 용도라면 우선 과제입니다.
2. **실제 브라우저 QA 미완료** — 개발 환경에서 실제 브라우저 구동이 불가해, **픽셀 단위 쪽나눔·두 면 보기·랜딩 무스크롤 맞춤·애니메이션**은 구조적 검증(Node/jsdom)만 했습니다. 인계 후 **크로스 브라우저·실기기 QA**가 필요합니다.
3. **브라우저 최소 버전** — `color-mix()`, `:has()`를 사용합니다. 대략 **Chrome/Edge 111+, Safari 16.4+, Firefox 121+** 이상이 필요합니다(구형/IE 미지원). 더 넓은 지원이 필요하면 해당 CSS를 대체해야 합니다.
4. **설정 영속화 미구현** — 새로고침 시 설정이 초기화됩니다. 다운로드 배포본에서는 localStorage가 동작하므로, 설정 저장이 필요하면 손쉽게 추가 가능합니다.
5. **파서 코어 동기화** — 검증을 위해 파서 코어를 별도 모듈(`core.js`)로 추출해 쓰는 패턴을 사용했습니다. HTML 내장 코어와 모듈을 **항상 동일하게 유지**해야 합니다. 빌드 단계에서 코어를 HTML에 주입하거나, 코어를 외부 파일로 분리해 단일 출처로 만드는 리팩터링을 권장합니다.
6. **확장 지점** — 서버 원고 로딩(토큰/링크 공유 등)은 `loadManuscript(md)`에 붙입니다. 이때 API·CORS·인증이 새로 필요합니다.
7. **접근성** — 키보드 이동·포커스 표시는 있으나, ARIA·스크린리더 대응은 보강 여지가 있습니다.
8. **모바일** — 반응형이며 좁은/낮은 화면에선 특성 영역을 숨기고, 모바일에선 좌우 화살표 대신 스와이프·탭 영역을 씁니다. 실기기 확인이 필요합니다.

## 10. 검증(회귀 테스트) 방법

규격 적합성은 **파서 코어를 추출해 픽스처와 비교**하는 방식으로 검증해 왔습니다. 권장 절차:

1. `미리봄.html`의 인라인 스크립트에서 파서 코어를 추출해 모듈(`core.js`)로 둔다.
2. Node에서 `assemble(픽스처)` 결과(본문 HTML·특수 섹션 수·표/각주 수·장 수 등)를 기대값과 비교한다.
3. 레이아웃·시각 요소는 jsdom으로 "로드 무오류·구조 일치"까지만 자동화하고(픽셀은 불가), 실제 화면은 브라우저에서 확인한다.

> 참고: 위 절차에 쓰던 `core.js`와 검증 스크립트 패턴이 있습니다. 리포지토리 `test/`에 넣을 수 있도록 정리해 드릴 수 있으니 필요하면 요청하세요.
